# eRPC Python Infrastructure

This folder contains the Python implementation of the eRPC infrastructure.

The Python implementation of eRPC is fully compatible with the C/C++ implementation at the
protocol level. Also, the classes mirror those in the C++ infrastructure.

## Contents

[erpc](erpc) - Python package for eRPC infrastructure.

[setup.py](setup.py) - setuptools setup script that installs the erpc package.

[setup.cfg](setup.cfg), [README_Pypi.md](README_Pypi.md) - necessary files for the pypi support.

## Installation

Pip package installer needs to be installed in your Python version. Use following way for Python3 pip installation:

```sh
sudo apt-get install python3 python3-pip
```

To install the eRPC Python infrastructure from pypi:

```sh
pip install erpc
```

To install the eRPC Python infrastructure from local folder ([erpc/erpc_python](/erpc_python)):

```sh
pip install .
```

Once installed, you can access the infrastructure via a standard import statement.

```python
import erpc
xport = erpc.transport.SerialTransport("/dev/ttyS1", 115200)
client = erpc.client.ClientManager(xport, erpc.basic_codec.BasicCodec)
```

## Framed transport packet logging

The Python infrastructure can optionally log raw TX/RX packets from framed transports. This is useful when debugging serial, TCP, SPI or I2C based eRPC links without changing application source code.

Packet logging is disabled by default. It can be enabled with environment variables before starting an existing Python eRPC client or server.

Each packet record includes:

- timestamp and direction (`TX` or `RX`)
- transport class name
- complete framed bytes (`frame_hex`)
- framed transport header and payload bytes
- message length and CRC fields
- optional BasicCodec message header decode
- optional generated Python IDL decode

### Console logging

```sh
ERPC_PYTHON_PACKET_LOG=stdout python examples/matrix_multiply_python/matrix_multiply.py --client --serial /dev/ttyS1 --baud 115200
```

PowerShell example:

```powershell
$env:ERPC_PYTHON_PACKET_LOG = "stdout"
python examples\matrix_multiply_python\matrix_multiply.py --client --serial COM17 --baud 115200
```

### JSON file logging

```sh
ERPC_PYTHON_PACKET_LOG=erpc-uart.jsonl ERPC_PYTHON_PACKET_LOG_APPEND=0 \
    python examples/matrix_multiply_python/matrix_multiply.py --client --serial /dev/ttyS1 --baud 115200
```

PowerShell example:

```powershell
$env:ERPC_PYTHON_PACKET_LOG = "erpc-uart.jsonl"
$env:ERPC_PYTHON_PACKET_LOG_APPEND = "0"
python examples\matrix_multiply_python\matrix_multiply.py --client --serial COM17 --baud 115200
```

The default file format is strict JSON Lines, with one compact JSON object per packet. This is the recommended format for tools and post-processing scripts.

Set `ERPC_PYTHON_PACKET_LOG_PRETTY=1` to write multi-line, indented JSON records for manual inspection:

```sh
ERPC_PYTHON_PACKET_LOG=erpc-uart.jsonl ERPC_PYTHON_PACKET_LOG_PRETTY=1 python your_client.py
```

Pretty output is easier to read, but it is not strict one-record-per-line JSONL.

### Generated Python IDL decode

Packet logging can use Python files generated by `erpcgen` to enrich BasicCodec logs with service and method names. When the generated `server.py` and `client.py` modules are importable, the logger also performs a best-effort decode of invocation arguments and matching reply reference outputs.

Use `ERPC_PYTHON_PACKET_LOG_IDL` to provide one or more generated Python package or module names. Items can be separated with commas or semicolons. The modules must be importable through `PYTHONPATH` or the current Python environment.

```sh
PYTHONPATH=erpc_python:examples/matrix_multiply_python \
ERPC_PYTHON_PACKET_LOG=erpc-uart.jsonl \
ERPC_PYTHON_PACKET_LOG_IDL=service.erpc_matrix_multiply \
    python examples/matrix_multiply_python/matrix_multiply.py --client --serial /dev/ttyS1 --baud 115200
```

PowerShell example:

```powershell
$env:PYTHONPATH = "erpc_python;examples\matrix_multiply_python"
$env:ERPC_PYTHON_PACKET_LOG = "erpc-uart.jsonl"
$env:ERPC_PYTHON_PACKET_LOG_IDL = "service.erpc_matrix_multiply"
python examples\matrix_multiply_python\matrix_multiply.py --client --serial COM17 --baud 115200
```

IDL decode failures are reported inside the packet log record and do not affect RPC transport behavior. Reply payload decode requires the same logger instance to observe the matching invocation first, because the decoder tracks request arguments by service ID, request ID and sequence number.

### Environment variables

| Variable | Description | Default |
| --- | --- | --- |
| `ERPC_PYTHON_PACKET_LOG` | Enables packet logging. Use `stdout`, `stderr`, `console`, or an output file path. Unset or false-like values disable logging. | Disabled |
| `ERPC_PYTHON_PACKET_LOG_FORMAT` | Output format for file destinations: `jsonl` or `text`. Console destinations always use text output. | `jsonl` for files |
| `ERPC_PYTHON_PACKET_LOG_DECODE` | Enables BasicCodec message header decode. | Enabled |
| `ERPC_PYTHON_PACKET_LOG_IDL` | Comma- or semicolon-separated generated Python IDL packages or modules used for IDL-aware decode. | Disabled |
| `ERPC_PYTHON_PACKET_LOG_PRETTY` | Pretty-print JSON records. Useful for manual inspection, but not strict JSONL. | Disabled |
| `ERPC_PYTHON_PACKET_LOG_APPEND` | Append to file destinations when enabled. Set to `0` to overwrite. | Enabled |
| `ERPC_PYTHON_PACKET_LOG_MAX_BYTES` | Maximum number of frame bytes printed in text output. The JSON output always records the complete frame. | Unlimited |

False-like values are `0`, `false`, `no`, `off`, `disabled` and `none`. True-like values are `1`, `true`, `yes`, `on` and `enabled`.
